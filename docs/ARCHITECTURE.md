# Архитектура — CDP-US (US B2B)

Только USA. RF-система отдельно (см. [../SEGMENTATION.md](../SEGMENTATION.md)).

## Модель

```
ОДИН АККАУНТ (US-облако, мультитенант)
  signup → коннекторы на сайт → единая веб-консоль «все данные»
        │
   ЯДРО: CDP — first-party сбор → identity → единый профиль
        │      B2B-профиль: фирмографика · интент · поведение · соцсигналы
        │      профили питают каждый модуль
   ┌────┴───────┬──────────────┬──────────────┬───────────────┐
 Email        Соц-разведка   YouTube/видео   Автоматизация   Согласие/право
              (SMM-интент)                   (соц+мессенджеры) (CCPA/CPRA)
   └─────────── каждый = апселл по usage/seat ──────────────────┘
ОБЩИЙ СЛОЙ: ИИ-генерация · автоматизация (Temporal) · данные (PG/CH/Neo4j) · коннекторы
```

## Несущие ставки

1. **Мультитенантность — центральный инженерный фундамент.** Аккаунт → включает модули.
   OIDC, изоляция тенантов, RBAC, биллинг. Это то, чего нет ни в одном исходном движке.
2. **B2B-модель данных = фирмографика + интент** (компания, размер, индустрия, сигналы
   покупки), а не потребительская демография. Путь Clearbit/6sense/RB2B (IP→компания).
3. **Согласие — несущая зависимость, не апселл-полировка.** В US сбор чувствительных данных
   без consent-слоя = юр-экспозиция (CCPA/CPRA покрывают B2B-контакты с 2023). Consent живёт
   в фазе 1, а не в фазе 3. См. [COMPLIANCE-US.md](COMPLIANCE-US.md).

## Слои

- **platform/** — identity/аккаунт, мультитенант (OIDC), биллинг, единая консоль, коннекторы/SDK.
- **core-cdp/** — ingest-gateway → identity-graph → unified profile → сегменты/скоринг.
- **modules/** — email · social-intel · youtube · automation · consent (каждый tenant-scoped).
- **общий слой** — ИИ-генерация, движок автоматизации, хранилища данных.

## Преимущество US-сегмента над RF

Ограничение РФ-резидентности здесь **отсутствует** → свободно US-облако (AWS/Vercel/SES).
Самая тяжёлая боль RF-CDP (трилемма RAM/РФ-бокс/EE) в US не возникает: ставим полноценный
стек на нормальном хосте.
